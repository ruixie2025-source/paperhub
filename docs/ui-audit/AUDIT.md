# PaperHub 部署前产品与界面审查

## 审查范围

- 核心目标：导入论文、管理论文、查看详情、结构化分析和 AI 问答。
- 审查视口：桌面端 1440 x 1000，移动端 390 x 844。
- 审查方式：实际运行当前前后端，检查页面、接口、构建结果和本地数据目录。
- 证据限制：截图可以确认布局和可见状态，但不能单独证明完整键盘操作或屏幕阅读器兼容性。

## 流程步骤

1. 桌面端论文库首页（`01-home-desktop.png`）
   - 状态：功能可见，但信息密度和主次关系失衡。
   - 新增表单长期占据左栏，论文库被压缩，长标题和关键词在窄表格中形成拥挤换行。
   - 首页与独立页面重复提供完整 AI 问答，导致空间利用率偏低。

2. 移动端论文库首页（`02-home-mobile.png`）
   - 状态：主要操作可访问，但论文表格无法适配窄屏。
   - 表格横向超出视口，编辑和删除操作被截断；用户需要横向移动才能完成核心任务。
   - 手动新增表单位于论文列表之前，移动端需要滚动很长距离才能看到已有论文。

3. 桌面端 AI 问答页（`03-chat-desktop.png`）
   - 状态：基本可用，但页面过空，缺少论文范围、索引状态和引用提示。
   - 页面标题与卡片标题重复，聊天历史区的层级和对话感较弱。

4. 移动端 AI 问答页（`04-chat-mobile.png`）
   - 状态：布局没有明显溢出，但输入区和结果区仍然像一个普通表单。
   - 返回入口、页面标题和聊天容器可以进一步压缩，给回答内容留出更多可用高度。

5. 桌面端论文详情页（`05-detail-desktop.png`）
   - 状态：信息完整，但摘要过长且缺少清晰分区。
   - 元数据、摘要、PDF 操作和阅读器都在同一张长卡片中，扫描效率偏低。

6. 移动端论文详情页（`06-detail-mobile.png`）
   - 状态：存在阻断级响应式问题。
   - PDF 页面固定为 760px 宽，撑大整个详情页，导致标题、摘要和页面主体横向溢出并被截断。

## 已确认的优点

- 核心流程已经形成闭环：论文管理、PDF、结构化分析和问答都有明确入口。
- 中文文案基本统一，加载、错误、空状态和删除确认已经具备。
- 组件以 shadcn/ui 和 TailwindCSS 为基础，后续统一视觉不需要更换技术栈。
- PDF 阅读器采用懒加载，前端生产构建可以正常完成。

## 部署前必须修复

1. 修复移动端 PDF 阅读器固定宽度导致的全页横向溢出。
2. 为论文列表提供移动端卡片布局或可靠的横向滚动容器，确保编辑和删除始终可见。
3. 补充 Vite 的 `/paper-analyses` 代理。当前后端接口返回 200，但前端代理路径返回 404，页面因此显示“加载历史分析记录失败”。
4. 解决前端 `/chat` 页面与后端 `POST /chat` API 同路径冲突。当前从首页单页跳转可以进入问答页，但直接访问或刷新 `/chat` 会被代理到后端并返回 405。
5. 删除论文和替换 PDF 时同步清理旧 PDF、ChromaDB 索引和分析记录。当前数据库只有 3 篇论文、3 个 PDF 引用，但 `backend/uploads/` 中已有 12 个文件，存在明显残留。
6. 为 PDF 上传增加体积限制、内容类型和 PDF 文件头校验，避免一次性读取超大文件耗尽内存或磁盘。
7. 部署到公网前加访问保护。当前没有登录或权限控制，任何访问者都可以新增、修改和删除数据，并消耗 DeepSeek API 额度。
8. 明确生产运行目录和持久化目录，避免 SQLite、uploads 和 Chroma 因工作目录变化而写到错误位置。

## 视觉与体验升级建议

1. 建立统一应用外壳：顶部品牌、论文库、研究分析、AI 问答三个清晰入口，并显示本地索引或服务状态。
2. 将“导入 PDF”设为首页主操作；把手动新增改为按需打开的对话框，不再长期占据左栏。
3. 论文库改为宽主内容区：搜索、数量、批量分析放在同一工具栏，操作按钮使用 Lucide 图标与提示。
4. 桌面端保留紧凑表格，移动端使用论文卡片；关键词拆成独立标签，并限制长文本造成的布局变化。
5. 结构化分析采用可横向比较的表格，固定论文列；移动端改为逐篇展开，单元格详情继续使用对话框。
6. 首页只保留紧凑 AI 助手入口或最近问答，完整对话集中到独立页面，避免功能重复。
7. 详情页拆为“论文信息”和“PDF 阅读”两个清晰区域；摘要支持折叠，PDF 阅读器宽度随容器变化。
8. 使用克制的中性色作为主体，加入一组研究工具风格的青绿色状态色，减少当前纯灰界面的模板感。

## 可访问性风险

- `frontend/index.html` 当前为 `lang="en"`，与中文页面不符，页面标题仍为 `frontend`。
- 导入和替换 PDF 使用自定义标签按钮，需要补充明显的键盘焦点状态。
- 异步加载、导入、分析和错误提示没有 `aria-live`，辅助技术可能无法及时获知状态变化。
- 问答页同时存在两个“AI 问答”标题，层级重复。
- 移动端横向溢出会破坏放大和重排体验；修复后仍需进行一次真实键盘与屏幕阅读器检查。

## 推荐实施顺序

1. P0：代理、响应式溢出、文件和索引清理、上传保护、生产路径配置。
2. P1：首页信息架构、按需新增、响应式论文列表、统一导航和问答页重构。
3. P2：视觉令牌、状态色、图标、细节交互和可访问性状态提示。
4. 验证：前端 lint/build、后端编译与健康检查、桌面/移动截图回归、PDF 导入与删除闭环。

## 部署资源提醒

当前本地 Embedding 使用 `BAAI/bge-m3`，模型较大，PDF 导入还会同步执行解析、元数据提取、切块和向量化。2 核 4G 云服务器存在内存和响应超时风险；若保持当前模型，优先选择 4 核 8G，或在部署前单独规划更轻的 Embedding 方案和重新索引流程。
